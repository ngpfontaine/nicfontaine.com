BONKR

¶

Boilerplate - Opinionated (kinda), NaKed, & Responsive

Webpage Github Try Out

Bonkr is a normalized CSS boilerplate with a focus on style-agnostic usability, a simple layout system, and responsive flexibility. It's created as a starter-template with default component proportions & weight and some basic styling that won't conflict with yours, + some nice (and independant) features like dialogs and a loader.

You won't find any colours, embedded fonts, or navigation here — just a beautiful blank-slate for your content.

It's also the base stylesheet for this site.

Features

  • B&W with hover, focus, etc state changes
  • Helper classes for border-radius, display modes, width
  • rem based margin & padding system
  • Base layout styling for semantic elements
  • Basic loader overlay on onload event
  • Dialog modal with focus change, ESC to close, open/close btns

Zoom Scheduler

¶

Tech

  • Node
  • Express
  • Passport
  • Handlebars
  • Moment
  • Bluebird
  • Zoom RestAPI

A small session-scheduling webapp created to connect a group of learners, over a few-week timespan, to multiple Zoom accounts for simulation delivery. Without the need for a full database, sign-up contact and meeting infoinfo are entered into a JSON file organized by date.

Upon booking, a confirmation email is sent and a pre-session reminder is set. Booking emails contain cancellation links that clear the Zoom meeting entry, reminder email, and booking slot.

It's quick, light, and easy to re-skin for new clients' pilot deliveries.

Start Page

¶

I wanted a personalized start page with common links I use, search features, weather updates, and relevant hud stats like IP and time and naturally it had to be navigable w/o the use of a mouse.

You can randomly change the colour palette, change search sites, and navigate the link grid all using the keyboard. Gravity Forms Validation

¶

Unhappy with form validation on our landing pages at Element-360 I created an extension with JQuery, exclusively to extend the capabilities of Gravity Forms. In watching behaviour analysis recordings I found a large number of people clicking the CTA to immediately receive information. This results in a page reload with error messages appended to the required fields - not an optimal user experience.

My goals were:

  • Live validation/disable button send until form fulfilled
  • UI change from button click - of blank required fields
  • Validation check (logic and visual) on field blur
  • Email regex validation
  • Client-side validation, w/o AJAX
  • Simple, safisfying UX (people have a mote of tolerance on landing pages)

Field blur, focus, and change events check input validation and form completion, while adding and removing visual classes. Additionally, I implemented a button text update on send - as on slower networks, lag is generally perceived as a site bug and users will spam-click out of frustration.
